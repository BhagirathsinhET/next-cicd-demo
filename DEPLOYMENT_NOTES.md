## Required GitHub Secrets

Add these in **Settings → Secrets and variables → Actions** on your repository:

| Secret | Description | Where to find it |
|--------|-------------|------------------|
| `VERCEL_TOKEN` | Vercel API token | vercel.com → Account Settings → Tokens |
| `VERCEL_ORG_ID` | Vercel organisation ID | Run `vercel link` and check `.vercel/project.json` |
| `VERCEL_PROJECT_ID` | Vercel project ID | Run `vercel link` and check `.vercel/project.json` |

## Branch Setup


### `dev` → `staging` → `main` flow

1. Feature branches merge into `dev`
2. When ready for testing, merge into `staging` — triggers staging deploy
3. After QA, merge `staging` into `main` — triggers production deploy + creates GitHub Release


## First-Time Setup Checklist

- [ ] Add all required secrets listed above to GitHub repository settings
- [ ] Create `staging` branch (if using staging environment)
- [ ] Set up GitHub Environments: **Settings → Environments** — add `staging` and `production`
- [ ] Enable required status checks on protected branches
- [ ] Copy `.env.example` to `.env` on your server(s) and fill in values
- [ ] Test the CI workflow by opening a draft PR

## Deployment Target Notes

### Vercel

- Run `npm i -g vercel && vercel login && vercel link` locally to generate the project config
- The generated `.vercel/project.json` contains `orgId` and `projectId`
- Staging deploys create preview URLs; production deploys update the main domain
- Set environment variables in Vercel dashboard per environment
